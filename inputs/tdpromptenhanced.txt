TD Bank (United States) Account Funding Demo — Branded Demo Specification (ENHANCED)
===================================================================================

Build a branded demo for TD Bank (U.S. retail banking, https://www.td.com) showing an external account funding flow. The consumer links an external bank account using Plaid (Plaid Layer OR Plaid Link, depending on demo mode), then the backend runs Identity Match, Auth, a funding-amount capture step in the host app, and Signal Evaluate before showing instant funds approval.

Open with the use case narrative and close with a strong value proposition. Messaging priority remains: anchor on `inputs/plaid-value-props.md` and `docs/plaid-demo-script-value-propositions.md` (verbatim or closely adapted), supplemented only for API accuracy.

USAGE — `scripts/scratch/utils/prompt-templates.js` only extracts `[[SLIDE_OUTPUT_BEGIN]]`…`[[SLIDE_OUTPUT_END]]` when that content lives in the ingested file named **`prompt.txt`**. To run the pipeline with this spec: replace `inputs/prompt.txt` with this file, **or** keep `tdpromptenhanced.txt` as a source-of-truth copy and sync the slide markers (and any sections you want) into `inputs/prompt.txt` before `ingest` / `script`.

-------------------------------------------------------------------------------
WHY THIS FILE EXISTS (REVERSE-ENGINEERED FROM PIPELINE + LATEST BUILD)
-------------------------------------------------------------------------------

Observed gaps in prior runs:

1. **JSON sidebar drift** — The product spec asked for a “right-side JSON panel” on insight screens. The build system’s DOM contract uses **one global** `#api-response-panel` populated from JavaScript (`showApiPanel` → post-processed to `_showApiPanelStub`). Models sometimes inlined JSON inside `.insight-layout` (duplicate panels, QA noise) or hand-rolled `goToStep` branches that **do not** match the `API_DATA[id]` pattern `build-app.js` re-injects after stripping `showApiPanel(` calls — leaving the overlay empty or hidden.

2. **Recording fallback** — `record-local.js` injects `window._recordApiResponses` from `demo-script.json` and, after `goToStep`, calls `_showApiPanelStub(data)` when step IDs look like insight steps **and** data exists. That only helps if `_showApiPanelStub` exists and clears `display:none` on the panel. Manual preview without injection still needs a correct app-side implementation.

3. **Slides** — Earlier prompt text said “do not include raw JSON on slides,” which fights the goal of PowerPoint-ready technical slides. This file splits **exec** vs **technical** slide rules and ties technical slides to `templates/slide-template/` plus an explicit JSON preview column.

This enhanced spec aligns **product language**, **demo-script.json** (`apiResponse` blocks), **HTML build**, and **slide generation** so you do not hand-patch JSON rails after the build.

These rules mirror `scripts/scratch/utils/prompt-templates.js` and `scripts/scratch/scratch/build-app.js` post-processing.

### A) Exactly one JSON surface for API bodies

- Raw Plaid API JSON appears **only** inside the global overlay:
  - `#api-response-panel` with `data-testid="api-response-panel"`.
- **Do not** place `<pre>` JSON columns, “insight-right”, `*-json-panel`, or duplicate “Response JSON” divs **inside** Plaid insight step markup. The insight step shows narrative, scores, value lines — not the full payload.
- TD consumer steps: panel stays `display:none` (and collapsed/closed).

### B) Wiring pattern the build can repair

Implement API population so `build-app.js` can strip duplicate `showApiPanel(` calls and restore the insight branch:

1. Define a map keyed by **exact** `demo-script.json` step ids (copy-paste ids, no drift):

```javascript
var API_DATA = {
  'identity-match-insight': { response: { /* full POST /identity/match body */ } },
  'auth-insight':           { response: { /* full POST /auth/get body */ } },
  'signal-insight':         { response: { /* full POST /signal/evaluate body */ } }
};
```

2. Implement **`function showApiPanel(data)`** (build renames this to `_showApiPanelStub` and strips/repairs calls) that:
   - Targets `#api-response-panel` only.
   - Removes inline `display:none` / shows the panel, applies your `visible` / slide-in classes.
   - Renders `JSON.stringify(data, null, 2)` in `#api-response-content` (or equivalent), monospace **14–15px**.
   - Supports collapse/expand (chevron or “Hide JSON” / “Show JSON”) **without** moving JSON back into the step div.

3. Inside **`window.goToStep`**, after activating the step:
   - For every step id **not** in `API_DATA`, hide the api-response panel completely.
   - For ids in `API_DATA`, call **`showApiPanel(API_DATA[id].response)`** (single call — no parallel inline JSON UI).

4. **Mirror for the recorder** (redundant but safe):

```javascript
window._stepApiResponses = {
  'identity-match-insight': API_DATA['identity-match-insight'].response,
  'auth-insight':           API_DATA['auth-insight'].response,
  'signal-insight':         API_DATA['signal-insight'].response
};
```

5. **Insight step IDs** — Include the substring `insight` in Plaid API milestone step ids (e.g. `identity-match-insight`, `auth-insight`, `signal-insight`) so automated recording heuristics and humans can grep them consistently.

### C) Expand / collapse default for recording

- **Default for insight steps: panel expanded** when the step loads (presenters collapse during live talk tracks). Prior builds sometimes defaulted to collapsed, which reads as “missing JSON” in video and QA frames.
- TD screens: never show the panel expanded.

### D) Toggle control test IDs (unique per step)

Per `CLAUDE.md`, avoid duplicate `data-testid` values. Use step-scoped ids, e.g.:
- `api-sidebar-expand-btn-identity-match`
- `api-sidebar-expand-btn-auth`
- `api-sidebar-expand-btn-signal`

MANDATORY BACKEND API SEQUENCE (STORYBOARD AND NARRATION MUST MATCH THIS ORDER)
-------------------------------------------------------------------------------

After `public_token` exchange yields an `access_token` (and you have the selected `account_id` from Link onSuccess or accounts list), the demo's LOGICAL and SEQUENCED flow SHALL be:

  1. POST /identity/match — Confirm account ownership first
       - Compare KYC PII on file at TD (demo host) to identity on file at the linked institution.
       - Narration: ownership verification before moving money; nickname/typo tolerance.
       - Plaid insight screen: Identity Match. **Global** `api-response-panel`: full `/identity/match` JSON.
       - **demo-script.json**: this step MUST include `apiResponse: { "endpoint": "POST /identity/match", "response": { ... } }` — the build and recorder use it.

  2. POST /auth/get — Retrieve rails for ACH funding
       - Use `access_token`; pass `account_ids` scoped to the funding account when limiting scope.
       - Narration: Auth supplies ACH account and routing numbers for the funding leg.
       - Plaid insight screen: Auth. **Global** panel: full `/auth/get` JSON.
       - **demo-script.json**: `apiResponse` for `POST /auth/get`.

  3. HOST APP STEP (TD consumer UX, not Plaid branded)
       - Prompt the end user for the funding amount (USD) they want to move into their TD account.
       - Validate input (positive decimal, reasonable max for demo). Store as the amount passed to Signal.
       - No JSON panel on this screen. Side panels remain hidden.

  4. POST /signal/evaluate — ACH return risk for the specific funding amount
       - Required inputs include: `access_token`, `account_id`, `amount` as a USD decimal (NOT cents), and `client_transaction_id` (unique, max 36 chars).
       - Strongly recommended for model quality: `client_user_id`, `user` object (name, phone, email, address), `device` (ip, user_agent), and optional `ruleset_key`.
       - Plaid insight screen: Signal. **Global** panel: full `/signal/evaluate` JSON.
       - **demo-script.json**: `apiResponse` for `POST /signal/evaluate`.

  5. TD consumer UX — Instant approval outcome
       - After Signal `ruleset.result` is `ACCEPT` (happy path), return to TD-branded UI and show that funds are approved instantly / available per demo copy (no micro-deposits).

Do NOT present Auth before Identity Match in narration, `goToStep` ordering, or insight screen order. Link/Layer connection still happens before any of these backend calls.

Technical note (Plaid): `/identity/match` and `/auth/get` do not have a hard Plaid dependency order; this spec chooses ownership-first ordering for demo logic and sales narrative. `/accounts/get` may be used as needed to resolve accounts.

-------------------------------------------------------------------------------
SIGNAL SCORE DIRECTION (OFFICIAL SEMANTICS — DO NOT INVERT)
-------------------------------------------------------------------------------

Per Plaid Signal documentation (see https://plaid.com/docs/api/products/signal/#signalevaluate):

  - `bank_initiated_return_risk.score` and `customer_initiated_return_risk.score` are on a 1–99 scale.
  - HIGHER numeric score = HIGHER ACH return risk.
  - LOWER numeric score = LOWER return risk.

For the instant-funding happy path, use LOW risk scores (illustrative: single-digit to low teens) together with `ruleset.result: "ACCEPT"`. Values must be consistent with the JSON shown in the collapsible panel.

`ruleset.result` may be `ACCEPT`, `REVIEW`, or `REROUTE` — happy path uses `ACCEPT`.

DEMO MODES — PLAID LAYER HAPPY PATH AND PLAID LINK HAPPY PATH
-------------------------------------------------------------------------------

The demo application MUST support TWO distinct entry paths that are functionally equivalent from TD's perspective but showcase different Plaid connection UX:

  A) Plaid Layer mode (template-driven onboarding)
       - Token creation uses Layer session flow: `POST /session/token/create` with `user.client_user_id` and Dashboard `template_id` as required by your environment.
       - Response provides a Link token used to initialize the same Plaid Link client SDK on the frontend.
       - Document explicit URL or query parameter, e.g. `?connection_mode=layer` or route `/demo/fund/layer`.

  B) Standard Plaid Link mode
       - Token creation uses `POST /link/token/create` with `products` including at least `auth` and `signal`, and identity configured per below.
       - Document explicit URL or query parameter, e.g. `?connection_mode=link` or route `/demo/fund/link`.

Requirements:
  - Both modes MUST converge on the same post-connection sequence: exchange token, then Identity Match, Auth, amount entry, Signal, instant approval.
  - README or in-app dev banner (non-recorded): list the two URLs/parameters so producers can record Layer and Link versions without code changes.
  - Simulated UI (if not live Plaid) must reflect mode: Layer flow shows template-driven steps; Link flow shows standard Link steps appropriate to the recording script.

ASKBILL / PLAID DOCS TECHNICAL APPENDIX (AUTHORITATIVE FOR INTEGRATION DETAILS)
-------------------------------------------------------------------------------

1. Token exchange: `POST /item/public_token/exchange` -> `access_token`, `item_id`.

2. Optional: `POST /accounts/get` with `access_token` to list accounts and `account_id`s.

3. Identity Match: `POST /identity/match` with `access_token` and `user` PII object; optional account scoping. Identity product must be enabled via Link token configuration.

4. Auth: `POST /auth/get` with `access_token`; optional `account_ids`. ACH numbers in `numbers.ach[]`. Some FIs return tokenized account numbers — see Auth docs.

5. Signal: `POST /signal/evaluate` with `access_token`, `account_id`, `amount` (USD decimal), `client_transaction_id`; recommend `client_user_id`, `user`, `device`, optional `ruleset_key`.

6. Layer vs Link: Link uses `/link/token/create`; Layer uses `/session/token/create` + template; both initialize the Link SDK with a link token.

7. Post-funding lifecycle: For production patterns, consider `/signal/decision/report` and `/signal/return/report` when applicable — omit from short demo unless scripted.

  - Identity Match: https://plaid.com/docs/api/products/identity/#identitymatch
  - Auth: https://plaid.com/docs/api/products/auth/#authget
