PLAID DEMO PIPELINE — PROMPT TEMPLATE (HUMAN OR AI)
================================================================================

PURPOSE
-------
Use this file as a **fill-in-the-blanks spec** for any Plaid product demo. A human
or an AI should replace every `«...»` placeholder with concrete values, delete
optional sections that do not apply, and keep section headers so the pipeline and
reviewers can scan the doc.

HOW TO RUN THE PIPELINE
-----------------------
1. Copy your completed spec to **`inputs/prompt.txt`** (the orchestrator and
   script stage read that path), **or** merge your filled sections into the
   existing `prompt.txt`.
2. Add a line **`Brand URL: «https://...»`** near the top if you want
   `brand-extract` to pull host-app colors/logo (fintech customer site).
3. Keep approved messaging pointers: anchor on `inputs/plaid-value-props.md` and
   `docs/plaid-demo-script-value-propositions.md` where this template says so.
4. For slide generation, the block between **`[[SLIDE_OUTPUT_BEGIN]]`** and
   **`[[SLIDE_OUTPUT_END]]`** is extracted only when it appears inside the file
   ingested as **`prompt.txt`** (see `scripts/scratch/utils/prompt-templates.js`).

AUTHORITATIVE TECHNICAL RULES (DO NOT CONTRADICT)
-------------------------------------------------
- Repo root **`CLAUDE.md`** — DOM contract (`data-testid`, `goToStep`, Plaid Link
  single-step recording, Signal score semantics, design tokens, manual nav).
- **`inputs/plaid-link-sandbox.md`** — sandbox institutions, credentials, IDV persona.
- AskBill / Plaid docs — exact API field names and response shapes for JSON shown
  in the app.
- Product knowledge files in **`inputs/products/`** — preferred place for approved
  value props, accurate terminology, and AI research notes by product family.

**Recommended prompt scaffolding by product family:**

**Funding / Auth / Signal**
- Flow should usually name the logical order explicitly: Link -> token exchange -> Identity Match and/or Auth -> amount entry -> Signal -> approval.
- Reveal semantics: low risk score + approval outcome, ownership confirmation, or instant rails retrieval.
- Good insight screens: Identity Match, Auth, Signal.

**CRA Base Report**
- Flow should usually include: `/user/create` or equivalent setup -> CRA Link token -> Link -> report creation / ready beat -> Base Report retrieval -> underwriting summary.
- Reveal semantics: report availability, balances/inflows/outflows, ownership, account stability.
- Good insight screens: report requested / report ready, Base Report summary, ownership or cash-flow trend highlights.

**CRA Income Insights**
- Flow should usually include: `/user/create` -> Link token with `["cra_base_report", "cra_income_insights"]` -> Link -> report-ready beat -> `/cra/check_report/income_insights/get` -> underwriting summary.
- Reveal semantics: verified income understanding from the CRA report, readiness for underwriting, and key report-derived insights.
- Good insight screens: report requested / report ready, income insights summary, supporting report context or warnings if explicitly requested.
