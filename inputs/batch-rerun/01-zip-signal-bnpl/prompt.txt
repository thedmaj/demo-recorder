**User journey (one sentence):**
A Zip BNPL borrower's repayment is evaluated in real time by Plaid Signal, routing low-risk payments to cost-efficient ACH and flagging elevated-risk ones for debit — so Zip processes every repayment on the right rail.

**Story arc (short prose):**
Open with the BNPL repayment challenge: Zip runs millions of repayments monthly, but without real-time bank signal, routing to ACH is a bet — every returned payment costs money and damages customer trust. Enter Plaid Signal. When a repayment is initiated on Zip's platform, a `POST /signal/evaluate` call scores the transaction against 900+ bank signals in milliseconds. A score of 8 (low risk) routes directly to ACH — faster, cheaper. A score of 64 (elevated) recommends debit instead. Close with the quantified outcome: ACH expansion lowers per-payment cost while the Signal production deployment reduces return rates toward Zip's ~3% target.

**Insight presentation rules:**
- Signal API panel (`POST /signal/evaluate`) on step 6 (host step). Use `#api-response-panel`.
- Signal scores for ACCEPT scenario: `bank_initiated_return_risk.score: 8`, `customer_initiated_return_risk.score: 12`. Do NOT use scores above 20 for an ACCEPT decision.
- `ruleset.result` must be `"ACCEPT"` — never `"REJECT"` (not a documented value); use `"REROUTE"` for high-risk contrast if shown.
- Include 2–3 `core_attributes` in the API panel: e.g., `plaid_connections_count`, `days_since_first_plaid_connection`, `days_since_last_plaid_connection`.
- Do NOT add `reason_codes[]` — not a documented Signal field.
- Slides use Plaid slide design system per `templates/slide-template/`.
