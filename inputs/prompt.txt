Bank of America Account Funding Demo — Link and Fund in Seconds, Not Days
================================================================================

Host: Bank of America — U.S. retail banking
Canonical URL: https://www.bankofamerica.com
Brand URL: https://www.bankofamerica.com

User journey (one sentence):
A Bank of America customer funds their BofA checking account from an external
bank by connecting with Plaid (embedded Link), verifying ownership with
Identity Match, pulling ACH rails with Auth, entering the transfer amount,
running Signal Evaluate for return-risk, and getting instant funds approval.

Story arc (short prose):
BofA wants to remove friction from account funding without taking on ACH
return risk. Today, customers either wait days for micro-deposits or give up
after a clunky credential hand-off. Plaid enters at the "Link external bank"
CTA on the Transfers screen. Plaid's embedded Link gets the user to a
connected external account in seconds. Identity Match confirms the external
account belongs to the BofA customer of record. Auth returns ACH account and
routing numbers. The customer enters the amount to transfer, and Signal
Evaluate risk-scores that specific transaction. Low-risk users are approved
instantly; funds show available same-session with no micro-deposit wait.

Product-specific reminders:
- Signal: low score = low ACH return risk; happy path is ACCEPT with low
  scores (illustrative: `customer_initiated_return_risk.score: 12`,
  `bank_initiated_return_risk.score: 10`).
- Identity Match: strong match on name/address/phone despite formatting
  differences (e.g. "Michael Allen" vs "Michael A."). Show an email
  mismatch in the JSON to illustrate nuance — the ruleset still approves.
- Link: pre-step narration ends on the embedded widget; the Link step
  narrates inside the modal only.
- Every insight step id MUST contain the substring `insight`
  (`identity-match-insight`, `auth-insight`, `signal-insight`) so
  automated recording heuristics and humans can grep them consistently.

Compliance / user data:

record and the external bank's file on one field (email) and strong matches
on name/address/phone to showcase fuzzy matching.

CRITICAL UI REQUIREMENT — PLAID INSIGHT SCREENS (PRODUCT LANGUAGE)
-------------------------------------------------------------------------------

For EVERY behind-the-scenes Plaid-branded full-viewport insight screen that
corresponds to **POST /identity/match**, **POST /auth/get**, and **POST
/signal/evaluate**:

1. The host viewport shows Plaid insight chrome (dark navy, teal accent) and
   does NOT embed the raw API body in the main layout.
2. The **global** `#api-response-panel` overlay (right rail) is labeled
   clearly (e.g. "API response") and shows the JSON for that step only when
   that step is active.
3. The panel is collapsible (chevron / Show / Hide) and animates in
   (~400ms) consistently with the app.
4. The consumer BofA UX must never show raw JSON — no `<pre>` JSON columns
   inside host steps, no duplicate "insight-right" / "api-json-panel" /
   "raw-json" divs inside insight step markup.

Default for insight steps: the api-response-panel opens expanded on step
load so recording frames never read as "missing JSON". BofA host steps
always start with the panel collapsed / hidden.

-------------------------------------------------------------------------------
MANDATORY BACKEND API SEQUENCE (STORYBOARD AND NARRATION MUST MATCH THIS ORDER)
-------------------------------------------------------------------------------

After `public_token` exchange yields an `access_token` and `account_id`:

1. POST /identity/match — Confirm account ownership first.
2. POST /auth/get — Retrieve rails for ACH funding.
3. HOST APP STEP (BofA consumer UX) — amount entry in USD.
4. POST /signal/evaluate — ACH return risk for the specific amount.
5. BofA consumer UX — instant approval (Signal ACCEPT).

Do NOT present Auth before Identity Match in narration, `goToStep`
ordering, or insight screen order.

-------------------------------------------------------------------------------
SIGNAL SCORE DIRECTION (OFFICIAL SEMANTICS — DO NOT INVERT)
-------------------------------------------------------------------------------

Per Plaid Signal documentation:
- `bank_initiated_return_risk.score` and
  `customer_initiated_return_risk.score` are on a 1–99 scale.
- HIGHER numeric score = HIGHER ACH return risk.
- LOWER numeric score = LOWER return risk.

Happy path uses LOW risk scores (illustrative: single digits to low teens)
with `ruleset.result: "ACCEPT"`.
