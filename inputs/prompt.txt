================================================================================
Spring EQ — One-tap HELOC application to cash-flow-backed approval (Plaid Layer + Plaid Consumer Report: CRA Base Report + CRA LendScore)
================================================================================

**Host:** **Spring EQ** — home equity lender (HELOCs and home equity loans)
**Brand URL:** https://www.springeq.com

**User journey (one sentence):**
A homeowner starts a HELOC application, taps once to prefill their details with Plaid Layer, then connects their bank in a Plaid Consumer Report (CRA) session so Spring EQ can underwrite on 365 days of verified cash flow and a CRA LendScore — and pre-qualifies the line in minutes instead of a multi-day document chase.

**Story arc (short prose):**
Gregory Hudson wants a $75,000 HELOC against his home. Spring EQ's digital origination funnel loses applicants to long forms, so it leads with Plaid Layer: Gregory is recognized and shares prefilled name, email, phone, and property address with one tap. To underwrite, Spring EQ asks him to connect his bank with FCRA-permissible consent in a CRA Link session. Plaid generates the Consumer Report asynchronously. When it's ready, Spring EQ reads the **CRA Base Report** (365-day cash flow — healthy inflows, zero NSFs, stable balances) and a **CRA LendScore** (1–99, higher = lower default risk) as a complementary cash-flow risk signal, and pre-qualifies the line — a faster path to a confident decision. Compliance-safe language only ("can help," "typically," "up to").

**Product-specific reminders:**
- LendScore: 1–99, **higher = lower default risk = safer to approve** (do NOT invert; this is the opposite direction from Plaid Signal). Predicts 12-month non-mortgage default risk; here it complements the Base Report as one cash-flow risk signal in a home-equity decision — never the sole basis. Retrieved via `/cra/check_report/lend_score/get` after report-ready; not a Link product string.
- CRA reports retrieved by `user_id` (not access_token); retrievable for 24h.
- Layer returns unverified prefill (network data) — frame as "prefilled / one tap to share," NOT "verified identity." KYC would require chaining IDV (out of scope here).
- `products[]` is research-driven — use `linkTokenCreate.suggestedClientRequest` verbatim; do not hardcode.
- Compliance-safe claims only: "can help," "typically," "up to"; never "guarantee" / "eliminate." Avoid leaning on GTM percentage stats in narration.
- Happy path only — no declines/errors in the main flow.

**Compliance / user data:** FCRA consumer-report consent + permissible purpose (`EXTENSION_OF_CREDIT`); Layer prefill is consumer-permissioned but unverified; representative sandbox data.
